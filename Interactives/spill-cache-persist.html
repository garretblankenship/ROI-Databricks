Part 5 · Memory Management

When RAM runs out: spill, cache, persist, unpersist

Spark's first instinct is always to keep data in memory. When it can't, three things happen — none of which kill the job, but all of which can quietly destroy performance. This page walks through spill (when execution runs out of RAM), cache and persist (when you explicitly choose to keep data around), and unpersist (when you need to take it back).

SCENARIO · 01
When the shuffle outgrows the heap

Every Spark executor has a fixed amount of memory, split into two regions: storage (for cached data) and execution (for shuffles, joins, sorts, aggregations).

When a wide operation like groupBy needs more execution memory than the region holds, Spark doesn't crash. It spills the excess to local disk and keeps going.

Click Run query to watch a GROUP BY exceed the executor's execution region and start spilling.

The key idea Spill never kills the job — it just makes it slower. Disk is ~250× slower than RAM (Part 0 established that). So spill-heavy queries finish, but they finish at disk speed instead of memory speed.

Why this matters: Memory is the difference between a fast Spark job and a slow one. Spill is what happens when execution outgrows the heap — it's automatic, it's safe, and it's silently expensive. cache / persist let you keep a DataFrame in memory across multiple actions so you don't re-scan storage on every reuse. unpersist is how you take memory back when you're done — without it, cached DataFrames sit around eating storage memory and forcing LRU eviction of more useful data. Every long Databricks notebook should be auditing both what gets cached and when it gets released.
