Why distributed data systems exist

Before you can understand Spark, you need the four ideas it's built on: decoupled compute and storage, vertical scaling, horizontal scaling, and the memory hierarchy. Each one solves a problem that pure single-machine computing can't.

Pros of going vertical

  • Simple mental model. One machine, no distributed-systems complexity.
  • No network overhead. All data stays local — every read is at memory speed.
  • ACID transactions are easy. No coordination across nodes.
  • Cheap at small scale. A single server handles most workloads under ~100 GB.

Cons of going vertical

  • Hard physical ceiling. The biggest cloud instances top out at ~24 TB RAM.
  • Cost scales worse than linearly. 10× the specs costs 30–50× more.
  • Single point of failure. Machine dies → everything stops.
  • Downtime to upgrade. Adding RAM means rebooting.

Pros of going horizontal

  • No ceiling. Scale past what any single machine can offer — add more boxes until you're done.
  • Linear cost scaling. Each worker is the same price. No exponential premium for "bigger".
  • Cheaper than premium tiers. Commodity machines beat enterprise hardware on $/throughput at most workloads.
  • Fault tolerance. One worker dies; the rest keep going.

Cons of going horizontal

  • Network overhead. Workers must talk; bandwidth and latency become real costs.
  • Coordination complexity. Failures, retries, ordering — all on you (or your framework).
  • Diminishing returns. 2× workers ≠ 2× speed once coordination dominates.
  • Distributed-systems bugs. Partial failures, clock skew, split brain.

SCENARIO · 01
Compute and storage want different lives

For decades, a database server meant one box holding both the CPUs that did the computing and the disks that held the data.

That seems natural — until you realize compute needs and storage needs grow at totally different rates. A company might 10× its data over a year but only need 2× the compute. With coupled architecture, you'd buy 10× the servers anyway.

Why this matters: Spark is built on all four of these ideas. It runs as a horizontal cluster that decouples compute from storage (it usually reads from S3/HDFS), it works hard to keep active data in RAM, and it spills to disk only when memory runs out. Once these foundations click, every Spark concept — partitioning, executors, shuffles, caching — has a clear "why" behind it.
